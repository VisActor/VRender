# VRender 图形状态与动画重构期望文档

## 文档目的

本文档用于定义 VRender 在“图形状态与相关动画”方向上的重构期望、系统边界、关键场景与技术约束。

本文档不是最终实现方案，也不是详细架构设计。它的目标是为后续架构师与资深开发者的设计讨论提供统一的问题定义、目标约束和评价标准，避免后续讨论在术语、职责边界和性能目标上发生偏移。

---

# 第一部分：重构目标、场景与边界

## 1. 背景与问题

当前 VRender 已经具备以下几类相邻但职责不同的能力：

- 图元静态状态能力，例如 `states / currentStates / stateProxy`；
- Group/Theme 共享样式优化能力，用于减少重复样式构造与合并；
- 动画能力，包括普通属性动画、生命周期动画以及 morph 等专用过渡能力。

这些能力在功能上已经可用，但在“状态真值由谁管理”“状态与动画的职责边界是什么”“共享状态定义如何高性能复用”“图元如何对外暴露最终生效样式”这些问题上，还没有形成足够清晰且可扩展的统一模型。

本次重构要解决的不是某一个 API 的细节问题，而是图元展示语义的真值来源问题。

## 2. 核心目标

本次重构的核心目标如下：

- 图元自己管理自己的静态展示状态；
- 外部主要负责声明图元当前有哪些状态，而不是负责手工拼接状态样式；
- 状态定义、状态优先级、状态冲突裁决应具备统一模型；
- 共享状态定义应优先放在 Group/Theme/模板等共享层，避免在单图元上重复构造；
- 图元对外仍能稳定暴露“最终生效属性”，降低上层图表库和表格库的改造成本；
- 状态系统在几千级图元交互下应保持稳定，在几万级图元场景下需要有明确的性能兜底策略；
- 动画能力在新的解释下仍然成立，且不会破坏状态真值模型；
- morph 这类关键能力必须保留。

## 3. 重构范围

本次重构覆盖的对象是“静态展示状态”。

它包括：

- 图元交互状态，例如 `hover`、`selected`、`active`、`disabled`；
- 外部业务自定义状态，例如筛选、联动、反选、高亮、弱化等场景下的 `customPick`、`select-revert`、`blur` 等状态；
- 状态切换引发的展示过渡动画。

它不包括：

- `appear / update / exit / highlight` 这类动画生命周期状态本体；
- morph 能力本身的重构；
- 对现有 incremental draw 机制的整体重写。

## 4. 推荐方向

建议采用“状态引擎优先的内核重建”方向：

- 用新的状态解析内核定义图元状态真值；
- 保持对外兼容层，尽量降低上层 break change；
- 共享现有 Group/Theme 的高性能复用思想；
- 将批量状态调度视为独立的性能策略，而不是将状态系统直接嫁接到现有 incremental draw 语义上。

不建议首轮重构就将“状态系统、动画系统、渐进式调度系统”一并做成大一统重写，这会显著扩大讨论与落地范围。

## 5. 关键设计原则

- 状态真值属于图元，外部只声明状态，不手工维护最终状态样式；
- 状态定义以共享为主，实例以消费为主；
- 交互态与业务自定义态在模型上没有本质区别；
- 状态与动画生命周期分离；
- 默认语义优先保证一致性，性能策略作为可选增强；
- 支持能力不等于所有场景都走慢路径；
- 性能优化应以批量状态切换为核心，而不是把状态解释逻辑带入渲染热路径。

## 6. 关键场景

本次重构至少需要覆盖以下场景：

- 单图元交互状态切换，例如 hover、selected、click；
- 多图元联动状态切换，例如多图表联动高亮、刷选、反选；
- 大规模状态切换，例如一个交互使成千上万个图元进入 `selected / select-revert / blur`；
- 状态切换时仅影响视觉属性；
- 状态切换时影响几何属性，例如路径、尺寸、位置等；
- 状态切换伴随动画过渡；
- 图元存在自驱动画时，状态仍然有明确真值；
- 业务显式调用 morph 时，状态真值与 morph 结果不冲突；
- 共享状态定义在 Group/Theme 侧统一配置，图元只负责声明当前状态。

## 7. 非目标

本次重构不以以下目标为第一优先级：

- 将所有动画能力并入状态系统；
- 在内核中引入状态槽位模型；
- 自动跟踪外部业务上下文依赖；
- 自动猜测是否应启用批量分帧状态提交；
- 在首轮实现中提供完全通用的实例级状态覆写体系。

## 8. 兼容性目标

兼容性目标如下：

- 对外仍尽量保留图元“最终生效属性”的读取习惯；
- 现有 `addState/removeState/toggleState/useStates/clearStates` 这类状态操作入口尽量延续；
- 上层图表库与表格库在迁移后，不需要全面重写其状态驱动方式；
- 不再保证 `graphic.attribute.xxx = value` 这种直接赋值写法的兼容；
- 正式写语义统一收敛到 `setAttribute / setAttributes`。

# 第二部分：技术方向与关键约束

## 9. 状态模型

### 9.1 平铺状态模型

内核采用无槽位的平铺状态模型。

所有状态一视同仁，包括：

- 交互状态；
- 业务自定义状态；
- 反选、高亮、弱化等衍生状态。

不在内核中引入 `interaction / selection / business` 这类槽位实体。

### 9.2 状态定义的最小单元

建议每个共享状态定义至少包含以下信息：

- `name`
- `priority`，默认值为 `0`
- `rank`，编译后的稳定顺序
- `patch` 或 `resolver`
- `exclude[]`
- `suppress[]`

### 9.3 状态集合

图元实例至少应暴露两组只读状态集合：

- `activeStates`
  表示真实激活状态集合；
  已完成归一化与排序；
  包含被 `suppress` 的状态；
  不包含被 `exclude` 移除的状态。

- `effectiveStates`
  表示真正参与样式求值的状态集合；
  它是 `activeStates` 的子集；
  不包含被 `suppress` 的状态。

### 9.4 集合编辑语义

现有以下 API 在语义上都应收敛为对目标状态集合的编辑：

- `addState`
- `removeState`
- `toggleState`
- `useStates`
- `clearStates`

它们的内部语义建议统一为：

1. 编辑目标状态集合；
2. 归一化；
3. 排序；
4. 生成 `activeStates`；
5. 计算 `effectiveStates`；
6. 解析最终状态结果。

## 10. 优先级与冲突语义

### 10.1 排序规则

状态顺序不再由调用方传入顺序决定，而由内核统一决定。

建议排序规则为：

1. 先按 `priority`
2. 再按编译后的 `rank`

这样可保证：

- 状态集合语义与传入顺序解耦；
- 同一组状态在不同调用来源下结果一致；
- 读取到的 `activeStates` 与 `effectiveStates` 都是稳定有序集合。

### 10.2 `exclude` 与 `suppress`

需要严格区分两类冲突语义：

- `exclude`
  表示状态迁移语义；
  目标状态从最终 `activeStates` 中被移除。

- `suppress`
  表示渲染压制语义；
  目标状态保留在 `activeStates` 中，但不进入 `effectiveStates`。

### 10.3 冲突裁决顺序

`exclude` 与 `suppress` 的裁决顺序建议与样式覆盖顺序一致：

- 高优先级状态对低优先级状态生效；
- 若优先级相同，则由 `rank` 决定；
- 若两个状态互相声明冲突，也不引入额外异常路径，仍由 `priority + rank` 裁决。

### 10.4 不自动恢复

当某状态被 `exclude` 移出 `activeStates` 后：

- 后续如果排斥方消失；
- 被排斥状态不自动恢复；
- 业务若仍需要它，必须重新显式设置。

这样可以避免内核额外维护 `requestedStates / effectiveStates` 两层持久状态真值。

## 11. 状态定义归属与共享复用

### 11.1 共享定义优先

状态定义应主要挂载在共享层，例如：

- Group
- Theme
- 模板或上层封装层

图元实例主要持有：

- 当前状态集合；
- 当前解析结果；
- 少量局部缓存。

不应在单图元上重复持有完整状态定义。

### 11.2 Group/Theme 复用方向

现有 Group/Theme 已具备共享样式复用与原型链继承优化思路。

重构后应尽量复用以下方向：

- 共享定义编译一次；
- 图元实例消费编译结果；
- 避免对每个图元重复 merge 完整状态定义。

### 11.3 实例级局部差异

不建议将“实例级同名状态覆写”作为核心路径。

对于少数图元的局部特殊逻辑，建议通过 `state resolver` 逃生口解决，而不是让状态来源同时分布在：

- 共享状态定义层；
- 图元实例覆写层；
- resolver 层。

## 12. `state resolver` 约束

### 12.1 单状态 resolver

建议只保留“单状态 resolver”：

- `resolver(stateName, activeStates, graphic, context) => patch`

不建议将“整组 activeStates -> 最终样式”的黑盒 resolver 作为核心能力。

原因是整组 resolver 会让：

- 优先级；
- `exclude/suppress`；
- 样式合并；
- 快慢路径判断

全部退化为业务黑盒逻辑。

### 12.2 可读取业务上下文，但不自动追踪

resolver 可以读取图元外部任意业务上下文，但 VRender 不负责自动探测外部上下文变化。

resolver 重新执行只由以下事件触发：

- 图元状态集合变化；
- 共享状态定义变化；
- 业务显式触发状态失效/重解析。

### 12.3 不进入渲染热路径

resolver 只能在以下阶段执行：

- 状态变化时；
- 显式失效时；
- 必要的初始化或重建阶段。

resolver 不能进入：

- render tick；
- 动画 tick；
- pick 热路径；
- bounds 热路径。

## 13. 属性分层与对外属性语义

### 13.1 内部属性分层

内部建议至少区分三层概念：

- `baseAttributes`
  业务通过 `setAttribute / setAttributes` 写入的基础属性真值；

- `resolvedStatePatch`
  当前 `effectiveStates` 解析出的合并结果；

- `finalAttributes`
  当前最终生效属性视图；
  语义上等于 `baseAttributes + resolvedStatePatch`。

### 13.2 对外兼容语义

对外建议继续保留“`graphic.attribute` 表示最终生效属性”的习惯，以降低 break change。

但其语义应调整为：

- 最终属性视图；
- 只读读取入口；
- 不再作为主写入对象。

### 13.3 写语义统一收敛到 base 层

`setAttribute / setAttributes` 的正式语义为：

- 始终写入 `baseAttributes`；
- 然后重新解析状态；
- 再得到新的 `finalAttributes`。

如果某个属性当前被状态覆盖，则：

- 该次写入会先沉在 base 层；
- 等状态不再覆盖该属性时，再显现出来。

### 13.4 不再兼容直接属性赋值

重构后不再保证以下写法的兼容：

```ts
graphic.attribute.fill = 'red';
```

直接赋值不再作为可靠写接口。

## 14. 状态解析产物与缓存策略

### 14.1 图元级缓存

图元长期持有的状态相关缓存建议至少包括：

- 当前 `activeStates`
- 当前 `effectiveStates`
- 当前 `resolvedStatePatch`
- 必要的脏标记、版本号与少量辅助元信息

### 14.2 不缓存每个状态的图元级副本 patch

不建议让每个图元长期持有“每个状态各自的一份独立 patch 副本”。

尤其在以下场景下会带来明显内存和 GC 风险：

- `selected / select-revert`
- 大规模联动高亮
- 多图表同步状态切换

共享静态 patch 应由共享层持有，图元只保存当前最终结果和必要的局部缓存。

### 14.3 resolver 结果缓存

如果某状态使用 resolver：

- 仅对当前图元当前活跃状态按需缓存结果；
- 当状态集合变化、共享定义变化或业务显式失效时作废；
- 不应将其作为长期稳定主数据。

## 15. 性能目标与快慢路径

### 15.1 性能优化焦点

本次状态系统性能优化的重点应放在“状态切换”阶段，而不是渲染热路径。

状态切换时可以接受有限的：

- 解析；
- 排序；
- patch 生成；
- 差异判断；
- 提交。

但 render、pick、bounds、动画帧中不应重复解释状态定义。

### 15.2 共享状态定义编译

共享状态定义应在初始化或更新时完成编译，编译结果至少包括：

- `priority`
- `rank`
- `exclude/suppress`
- 是否包含 resolver
- patch 涉及的 key 集合
- key 的影响分类

图元在状态变化时只消费编译产物。

### 15.3 自动推断影响范围

状态系统必须自动推断本次状态变化的影响范围，不要求业务显式声明。

自动推断基础来自：

- 编译后的状态定义；
- 本次真实 patch delta。

不建议将这部分复杂度上抛给业务。

### 15.4 建议的影响分类

状态变更建议至少区分以下影响分类：

- `paint`
- `transform`
- `shape`
- `bounds`
- `layout`

具体命名可以调整，但语义分层必须存在。

### 15.5 纯视觉快路径

如果本次状态切换未触及几何相关 key，则应保持纯视觉快路径。

这意味着：

- 状态系统可以支持几何状态；
- 但只有当状态 patch 真实改动几何属性时才进入慢路径；
- 未使用几何状态时，性能目标应尽量接近纯视觉状态系统。

### 15.6 几何状态允许存在，但不应拖慢默认路径

本次重构不应简单以“彻底禁止几何状态”换取性能。

更合理的设计是：

- 核心支持几何状态；
- 自动区分快慢路径；
- 让“能力存在”不等于“所有场景都走慢路径”。

### 15.7 状态提交不能直接复用当前普通属性更新语义

当前普通属性写入路径即使仅更新视觉属性，也可能触发较重的更新标记。

新的状态内核应采用：

- 基于最终 patch delta 的差异分析；
- 基于影响分类的精确 tag 提交；
- 尽量避免无差别走全量属性更新语义。

### 15.8 diff 应基于最终结果，而不是仅基于状态列表变化

状态列表变化不等于最终渲染结果变化。

例如：

- `hover + selected`
- 切换为 `selected`

若最终生效属性无变化，则不应付出不必要的更新代价。

因此提交判断应基于：

- 旧 `finalAttributes` / 旧状态 patch；
- 新 `finalAttributes` / 新状态 patch；
- 它们之间的真实 delta。

## 16. 状态与动画的边界

### 16.1 三类概念分离

建议明确区分三类能力：

- 静态状态
- 状态过渡动画
- 自驱动画

静态状态定义“图元此刻应该长什么样”；
状态过渡动画负责“状态变化时如何平滑过渡”；
自驱动画负责“动画逻辑自身暂时接管某些属性”。

### 16.2 静态状态是真值源

无论是否存在动画，图元最终“应该是什么样”，都由：

- `baseAttributes`
- 当前状态集合
- 当前状态解析结果

共同决定。

### 16.3 状态过渡动画

状态过渡动画服务于静态状态切换。

因此：

- 它不是新的真值源；
- 当目标状态再次变化时，其目标应随之更新、重启或中断；
- 它始终服从新的状态真值。

### 16.4 自驱动画

自驱动画由动画逻辑自己定义目标属性和时间过程。

建议默认语义为：

- 自驱动画运行期间可临时优先；
- 但只是“临时接管”；
- 动画结束后，如其影响的属性受状态系统管理，则控制权回到当前状态解析结果。

自驱动画结束后，不默认把动画终点固化为新的真值。

### 16.5 动画与状态冲突策略

建议在文档层明确：

- 状态过渡动画默认服从状态真值；
- 自驱动画可在配置层声明优先级或控制权策略；
- 状态与动画可共享最终属性视图，但不能共享 source of truth。

## 17. Morph 的保留方式

Morph 不能被抛弃，但不建议并入静态状态系统。

建议继续保持：

- 作为业务显式调用的专用几何过渡能力；
- 主要用于图表类型切换、形状切换等强语义过渡场景；
- 由上层图表逻辑主动触发。

在新的解释下，Morph 需要满足：

- 运行期间不破坏状态真值模型；
- 结束后图元仍可回到当前 `base + state` 的一致结果；
- 不被错误地视为“状态定义的一部分”。

## 18. 批量状态切换与渐进式调度整合

### 18.1 默认同步收敛

默认状态语义仍然是同步收敛：

- 普通规模状态切换应在一次提交中完成；
- 保证图元状态、最终属性、渲染和交互一致。

### 18.2 可选的分帧状态提交

针对大规模交互场景，应支持可选的延迟/分帧状态提交策略。

该策略不是默认语义，而是：

- 性能兜底能力；
- 业务可显式启用的策略。

### 18.3 与现有 incremental draw 的关系

现有 incremental draw 主要解决的是：

- incremental group 内容在独立 subLayer 中逐帧绘制。

而批量状态切换解决的是：

- 对已存在的大量图元逐步完成状态重解析、提交与重绘。

建议整合方式为：

- 复用现有渐进式渲染的调度与区域级基础设施；
- 但引入独立的 `state batch job / deferred state update` 任务模型；
- 不将批量状态更新直接等同于现有 incremental draw 语义。

### 18.4 配置层级

分帧状态提交建议配置在以下层级：

- Stage
- Layer
- Group

它更像区域级渲染策略，而不是单次 `useStates` 调用的参数。

### 18.5 仅默认适用于纯视觉状态

分帧状态提交默认只适用于 `paint-only` 状态变化。

如果状态变化涉及：

- 几何
- bounds
- layout
- pick 一致性

则默认仍同步收敛。

这样可以避免多帧收敛窗口内：

- 视觉与命中测试不一致；
- 包围盒与布局关系失真；
- 调试复杂度显著上升。

### 18.6 多帧收敛窗口

当业务显式启用分帧状态提交后，可以接受在短暂多帧窗口内：

- 部分图元已完成新状态提交；
- 部分图元仍保持旧状态；
- 最终在若干帧内完全收敛。

这是策略性折中，而不是默认行为。

### 18.7 优先目标

分帧状态提交的首要目标不是绝对最短总耗时，而是：

- 避免主线程长时间阻塞；
- 保持用户交互响应；
- 将大规模状态切换成本分摊到若干帧中。

# 第三部分：重点关注事项

## 19. 后续架构与实现讨论中的重点关注项

以下内容应被视为后续架构设计与实现阶段的重点关注项，而不是附属优化项。

### 19.1 性能

性能是本次重构最重要的关注点之一。

后续方案必须优先回答以下问题：

- 纯视觉状态切换是否仍然是快路径；
- 几千级图元交互是否足够稳定；
- 几万级图元状态联动是否有明确性能兜底方案；
- resolver、patch 合并、状态排序和状态 diff 是否会带来不可接受的对象分配与 GC 压力；
- 共享状态定义是否真正减少了重复构造与重复 merge。

### 19.2 稳定性

稳定性与正确性是本次重构的另一项核心关注点。

后续方案必须保证：

- 状态真值、动画真值、最终属性真值之间不存在歧义；
- 状态切换、动画结束、morph 完成后都能回到一致结果；
- `exclude/suppress` 的行为可预期；
- 批量状态切换、延迟状态提交等策略不会引入难以定位的随机行为；
- 默认语义优先保证可推理性，而不是隐藏复杂度。

### 19.3 可调试性

重构后系统必须便于定位问题，尤其是在复杂交互场景下。

建议后续实现重点关注：

- 是否能清晰读取 `activeStates` 与 `effectiveStates`；
- 是否能分辨某个状态为什么存在但未生效；
- 是否能区分当前结果来自 base、状态还是动画临时接管；
- 是否能在大规模联动与批量更新场景中快速定位性能瓶颈。

### 19.4 兼容成本

虽然本次允许对内核进行较大重构，但仍应控制上层迁移成本。

后续设计应重点评估：

- 对图表库与表格库的接入成本；
- `graphic.attribute` 读取语义调整后的兼容策略；
- 旧状态 API 的延续与兼容包装；
- 动画与状态重构后，上层代码是否仍能保持清晰调用关系。

### 19.5 范围控制

本次重构要解决的是“图元静态状态真值模型”。

后续讨论中需要警惕范围失控，尤其要避免：

- 将动画生命周期状态重新并入静态状态系统；
- 将 morph 逻辑强行内聚到状态内核；
- 在首轮设计中引入过多高级实体，例如槽位系统、自动依赖追踪、实例级复杂状态覆写体系；
- 因追求抽象完整性而牺牲实现难度、性能和可迁移性。

---

## 20. 结论

本次重构的推荐方向是：

- 以图元静态状态真值模型为核心；
- 以共享状态定义、高性能状态解析与明确的动画边界为约束；
- 在默认一致性语义之上，提供可选的大规模状态调度策略；
- 将性能、稳定性和可调试性作为第一优先级进行架构与实现讨论。

后续完整架构方案应围绕本文档中的边界、目标与约束展开，而不是重新回到“状态只是样式 patch”或“动画与状态混合解释”的旧模型。
