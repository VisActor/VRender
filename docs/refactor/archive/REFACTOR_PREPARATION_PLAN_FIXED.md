# VRender 重构准备工作方案（修订版）

## 1. 背景

这份文档用于替代现有过于泛化的重构前准备计划，聚焦当前已经明确的两条重构主线，并保留讨论过程中确认的重要上下文，作为后续继续细化方案、收敛问题、制定实施计划的基础文档。

当前结论是：这次不是全仓泛重构，也不是单独的动画模块重构，而是一次局部但结构性较强的重构准备工作。

## 2. 当前重构目标

本次重构聚焦两个方向：

1. 删除当前基于 `inversify / inversify-lite` 的依赖注入体系
2. 重构一条跨模块链路：
   `graphic state model -> state style resolution -> state transition orchestration -> animation execution`

这里需要特别保留一条讨论结论：

- “图形状态 / 样式 / 状态切换触发动画”的混乱逻辑不完全属于动画模块
- 根因之一在于当前 graphic 层本身没有一套完整、稳定、可推导的状态样式逻辑
- 因此这次不应被定义为“动画模块重构”，而应定义为“图形状态系统重构，动画只是执行层之一”

## 3. 兼容性边界

已确认采用下面这档兼容性目标：

- 公开 API 签名和主要入口尽量保持不变
- 允许修正当前不一致、混乱或错误的行为
- 行为变化需要通过设计说明和迁移说明明确记录

这意味着：

- `stage`
- `components`
- 面向用户的主要 `graphic` 使用方式

原则上不进行大面积破坏性接口改造，但内部结构可以为长期稳定性做较大调整。

## 4. 对依赖注入重构方向的决策记录

针对“删除 inversify”这件事，讨论过两个长期方向：

### 方案 A

- 彻底弱化容器式 DI 模式
- 改成显式注册 + 工厂/注册表 + 普通构造传参

### 方案 B

- 保留一部分注册表式扩展点
- 去掉装饰器和隐式注入
- 作为较平滑的中间过渡形态

最终结论：

- 长期目标架构选择方案 A
- 也就是说，不只是移除外部依赖或本地 `inversify-lite` 实现
- 而是连同“容器式依赖组装”这套思路一起退场

保留的判断依据如下：

- 当前问题不只是实现重，而是依赖关系被装饰器、模块绑定、容器解析隐藏，导致阅读和重构成本高
- `render / picker / env contribution / graphic factory` 这类能力更适合显式注册和能力表，而不是隐式注入
- 从长期结构稳定性看，显式组装比容器解析更容易理解、验证和演进

同时也保留一条重要判断：

- “保留注册表式扩展点”可以作为迁移手段
- 但不应作为终局架构目标

## 5. 目标架构草案

### 5.1 对象组装轴

这一条轴用于替代当前基于 `inversify / inversify-lite` 的对象组装和扩展注册机制。

建议拆成四类核心角色：

#### 1. Factory

负责创建核心对象，例如：

- `stage`
- `layer`
- `graphic`
- render service
- picker service

#### 2. Registry

负责注册可扩展能力，例如：

- graphic renderer
- picker
- env adapter
- plugin
- custom contribution

#### 3. Plugin API

负责以显式方式安装一组注册行为，而不是通过 `ContainerModule` 在运行时隐式 bind。

#### 4. Composition Root

在少数入口文件集中完成系统装配，例如：

- browser 环境入口
- node 环境入口
- wx / lynx 等平台入口
- 默认 core 初始化入口

这一条轴的原则是：

- 运行时对象依赖必须能从构造参数、工厂参数或 registry 查询直接看出来
- 不再依赖装饰器、注入元数据、容器扫描、service identifier 解析
- “可插拔”能力保留，但实现方式改为显式注册

### 5.2 状态执行轴

这一条轴用于替代当前 graphic state 与 animation 互相穿透、职责混叠的实现方式。

建议拆成五类核心角色：

#### 1. State Model

定义 graphic 当前有哪些状态，以及：

- 状态优先级
- 状态是否互斥
- 状态是否允许叠加
- 状态集合的规范表示

#### 2. State Style Resolver

负责将下面这些输入解析成最终目标样式：

- `normal attrs`
- `states`
- `stateProxy`
- 未来可能的自定义 resolver

这一层应尽量纯函数化，不直接执行动画，不修改图形运行时。

#### 3. State Transition Orchestrator

负责判断：

- 从旧状态切到新状态时，哪些属性直接跳变
- 哪些属性进入 transition
- 哪些属性禁止动画
- 清状态和叠加状态时如何处理回退值

#### 4. Animation Executor

只负责执行 transition 或独立动画，不再负责定义状态语义。

也就是说，动画层应该回答：

- 如何执行
- 何时完成
- 如何中断

而不是回答：

- 当前应该应用哪个状态
- 目标样式应该是什么

#### 5. Graphic Runtime Hooks

graphic 层保留最小入口，例如：

- `useStates`
- `addState`
- `removeState`
- `clearStates`

但内部不再自己负责：

- 拼装状态样式
- 维护复杂回退值逻辑
- 直接编排状态动画

graphic 应该是状态变更的发起者，而不是状态求值器和动画编排器。

## 6. 对当前问题的结构性判断

根据当前仓库代码形态，已经确认以下问题：

### 6.1 DI 问题不是单点问题

仓库中不是简单依赖了一个外部 `inversify` 包，而是已经内嵌了一整套本地化实现：

- `packages/vrender-core/src/common/inversify`
- `packages/vrender-core/src/common/inversify-lite`

业务代码广泛依赖其中的：

- `Container`
- `ContainerModule`
- `bind`
- 各类装饰器注入能力

所以“删除 inversify”本质上是一次对象组装模型重构，而不是一次简单的依赖替换。

### 6.2 状态系统问题不是动画模块独占

当前 graphic 公开接口已经直接暴露了状态能力：

- `states`
- `stateProxy`
- `useStates`
- `addState`
- `removeState`
- `applyStateAttrs`
- `reApplyAnimationState`

这说明当前“状态模型、状态样式求值、状态动画触发”并没有清晰分层，而是已经直接进入 graphic 核心对象。

进一步确认的结构问题是：

- graphic 内部同时承担状态解析、normalAttrs 维护、动画触发、finalAttribute 修补等职责
- animate 侧又通过状态扩展机制补了一层 animation state manager
- 两边形成双向侵入，导致职责边界不清

## 7. 当前阶段的准备工作目标

当前阶段不是立即进入实现，而是先围绕目标架构继续做设计澄清。

准备工作的目标应调整为：

1. 明确新的对象组装模型
2. 明确新的图形状态模型
3. 明确状态样式求值与动画执行之间的边界
4. 标出必须兼容的公开 API 与允许校正的行为
5. 在此基础上，再反推测试补充、验证策略和实施拆分

这意味着旧文档中那种“全包铺开的测试补充清单”不再作为当前优先级最高的驱动项。

新的优先级应是：

1. 目标架构澄清
2. 模块边界与职责澄清
3. 兼容性边界澄清
4. 针对本次重构范围补测试与验证

## 8. 后续文档扩展方向

这份文档之后应继续补充以下内容：

1. 现状调用链梳理
2. 目标模块边界图
3. 关键数据结构设计
4. 状态优先级与覆盖规则
5. 状态切换与动画编排规则
6. 旧 DI 体系到新组装模型的迁移路径
7. 测试与验证策略
8. 风险清单与回滚边界

## 9. 当前文档用途

这是一份第一版修订草案，作用是：

- 固化当前讨论已经确认的决策
- 纠正原有准备文档中过于泛化的问题
- 作为后续继续深入讨论和形成正式设计文档的起点

它不是最终版实施方案，也不是任务排期文档。
