## 📝 执行规范

### 测试编写规范
- 使用 **Jest + ts-jest**
- 遵循 **AAA 模式**：Arrange, Act, Assert
- 测试命名要描述性：`should apply single state correctly`
- 每个 test 独立，使用 beforeEach/afterEach
- 覆盖正常、边界、异常情况

### 测试文件位置

## 🐛 遇到问题？你的处理方式

### 测试失败
1. 阅读错误信息和堆栈跟踪
2. 查看被测试的源码实现（`packages/vrender-core/src/graphic/`）
3. 检查测试用例是否正确理解了被测试代码的行为
4. 添加调试信息（`console.log`）查看中间状态
5. 如果是被测试代码的 bug，在任务描述中记录

### 任务阻塞
1. 在 `TASK_QUEUE.md` 的任务描述中标记：`🚫 阻塞原因：...`
2. 描述清楚阻塞的具体原因
3. **跳过该任务，继续下一个可执行的任务**
4. 等待架构设计师或总监与协调者处理

### 需求不明确
1. 在 `TASK_QUEUE.md` 的任务描述中添加：`❓ 问题：...`
2. 描述清楚你的疑问
3. 做出你认为合理的假设并实现
4. 在完成标记中说明你的假设

### 实现困难
1. 查看现有测试文件作为参考（`packages/vrender-core/__tests__/`）
2. 阅读 `TASK_TEMPLATE.md` 中的示例代码
3. 阅读 `CLAUDE.md` 中的测试部分
4. 如果仍然困难，标记为阻塞并继续下一个任务

### 发现架构问题
如果你发现架构设计有问题（比如任务描述与实际代码不符）：
1. 在任务描述中添加：`⚠️ 架构问题：...`
