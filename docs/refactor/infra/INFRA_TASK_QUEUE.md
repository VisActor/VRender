## 🎯 任务目标

解决 jest-electron 偶发启动崩溃问题，建立稳定的测试基础设施。

### 问题描述
- Electron 测试运行器在启动阶段偶发崩溃
- 本轮观察到多次崩溃报告（Electron-2026-03-27-*.ips）
- 崩溃后重跑通常恢复，但严重影响开发效率
- 影响所有需要 Electron 环境的单元测试

---

## 📝 任务分解

### 任务 1：调查崩溃根因
**优先级**：🔴 P0
**预估时间**：1-2 小时
**当前状态**：✅ 已完成

**任务描述**：
调查 jest-electron 崩溃的根本原因。

**调查方向**：
- [ ] 分析崩溃日志（*.ips 文件）
- [ ] 检查 Electron 版本兼容性
- [ ] 检查 jest-electron 版本和配置
- [ ] 检查系统资源限制（内存、进程数）
- [ ] 检查并行测试配置（是否资源竞争）
- [ ] 检查 rush 并行执行配置

**输出**：
- 根因分析报告
- 崩溃触发条件和频率统计

**调查结果**：
- 根因：崩溃发生在 Electron 主进程启动窗口期，最新崩溃日志 `Electron-2026-03-27-173514.ips` 显示 `Electron 32.3.3` 在 `macOS 26.3.1` 上于 AppKit 初始化阶段 `SIGABRT`，堆栈稳定落在 `___RegisterApplication_block_invoke -> _RegisterApplication -> GetCurrentProcess -> [NSApplication sharedApplication] -> _NSInitializeAppContext`。
- 版本结论：历史旧链路中的 `Electron 11.5.0` 已确定不兼容；升级到 `@pixi/jest-electron@26.1.0` 后仍有偶发启动崩溃，但问题已收敛为启动链稳定性，而不是版本完全失效。
- 崩溃频率：修复前为偶发，表现为“首次启动可能崩溃，重跑通常恢复”；修复后在 `vrender-core` 上连续 10 次未复现。
- 触发条件：发生在 Electron 进程向 Jest 回报 `ProcReady` 之前，属于主进程初始化失败，不依赖具体测试逻辑。

**完成标记**：
- ⏸️ 待开始
- 🔄 进行中
- ✅ 已完成

### 任务 2：评估替代方案
**优先级**：🔴 P0
**预估时间**：1-2 小时
**当前状态**：✅ 已完成

**任务描述**：
评估不同的解决方案，选择最优方案。

**方案选项**：

#### 方案 A：修复当前 jest-electron
- 升级 Electron 版本
- 调整 jest-electron 配置
- 添加重试机制
- 限制并行度

#### 方案 B：迁移到 jsdom
- 移除 Electron 依赖
- 配置 jsdom 环境
- 修复浏览器 API mock
- 评估功能损失

#### 方案 C：使用 @vitest/browser
- 评估 vitest 迁移成本
- 配置浏览器测试环境
- 对比测试执行速度

#### 方案 D：Node.js 原生 Test Runner
- 评估 Node 20+ test runner
- 配合轻量级 DOM 库
- 评估工作量

**评估维度**：
- 工作量
- 稳定性提升
- 功能兼容性
- 执行速度
- 维护成本

**输出**：
- 方案对比文档
- 推荐方案及理由

**方案对比**：
| 方案 | 工作量 | 稳定性 | 兼容性 | 速度 | 建议 |
|------|--------|--------|--------|------|------|
| A：修复当前 jest-electron | 低 | 高 | 高 | 中 | 推荐 |
| B：迁移到 jsdom | 高 | 中 | 中低 | 高 | 不推荐立即执行 |
| C：使用 @vitest/browser | 高 | 高 | 中 | 高 | 不推荐立即执行 |
| D：Node.js 原生 Test Runner | 高 | 中 | 低 | 高 | 不推荐立即执行 |

**推荐方案**：方案 A，修复当前 `jest-electron`
**理由**：
- 现有测试体系已经依赖 Electron 环境，迁移会扩大改动面。
- 当前问题集中在启动窗口期，适合通过自定义 runner 为启动过程补齐超时、失败判定和重试。
- 方案可以先在 `vrender-core` 验证，再无缝推广到其他包，回滚成本也最低。
